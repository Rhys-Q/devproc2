## 1 nn.Module实现不规范
在设计规范中，nn.Module对外只有forward函数和forward_fast函数。forward 函数提供标准实现，forward_fast函数提供hpc高性能实现。
当前很多nn.Module不符合这个设计规范，需要改造。

## 2 cuda kernel 接入实现丑陋
当前在构图时，通过call_pi05_cuda_kernel函数做了一层warp。
预期不应该有这一层warp。应该直接：
``` python
dp.cuda_call(
            "kernels/add_one.cu::add_one",
            x,
            y,
            4,
            metadata={"grid": (1, 1, 1), "block": (64, 1, 1)},
        )
```

## 3 forward 函数缺少实现
当前python/devproc2/models/pi05/modules.py中很多forward函数直接调用了forward_fast。这是不被允许的。
forward 函数中应该使用标准op 或者调用其他nn.Module 中的forward函数实现。这是标准op路径。

一个nn.Module中，必须存在forward函数，可以没有forward_fast函数。
编译时应该有两种编译模式，一种是 normal 路径，这条路径只使用forward函数。一种是高性能路径，这种会优先使用nn.Module的forward_fast函数，只有当forward_fast函数不存在时，才会fallback到forward函数。
你需要领域这种设计哲学，帮我重构pi05模型的modules.py。而且要在nn.Module的实现中增加check逻辑，违反设计的写法都应该直接报错！
